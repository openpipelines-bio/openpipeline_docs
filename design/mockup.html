One run, many samples — the param_list

To process a whole cohort in a single command, pass a param_list: a list where each entry is one sample with its own arguments (id, input, and any per-sample thresholds). Arguments you set outside the list apply to every sample. So shared settings stay global, while per-sample overrides live next to their sample — no scripting, no repeated commands.

# params.yaml — two samples, per-sample filter + a shared setting
param_list:
  - id: sample_1
    input: data/sample_1.h5mu
    rna_min_counts: 2          # only for sample_1
  - id: sample_2
    input: data/sample_2.h5mu
    rna_min_counts: 1          # only for sample_2
rna_max_counts: 1000000          # applied to every sample

How it works

The concepts that make everything else make sense — read once, apply everywhere.

One data format: MuData

Every component reads and writes the same HDF5-based MuData object — modalities under .mod, cells in .obs, features in .var. That shared format is why components snap together without conversion.

Components & workflows

A module does one job (normalize, cluster). A workflow chains modules into a pipeline. Both are built with Viash and containerized, so a local run and a cloud run behave identically.

Built with Viash · run on Viash Hub

Viash turns a config + script into a containerized component and a Nextflow module. Viash Hub is the web catalog where those components live — browse them, read their parameters, and launch workflows through a GUI or connect to Seqera, without touching the CLI.

The pipeline model

Ingestion → Process samples → Integration → Downstream. Spatial data follows the same shape with spatial-aware steps; GPU and QC components slot into existing steps; turnkey pipelines wrap the whole thing.

Many samples at once: the param_list

A param_list processes a cohort in one run: each entry is a sample with its own arguments, while settings outside the list apply to all samples. See it in action in Get started →

Advanced: the data-state view

Under the hood, processing is organized by the state of the data (single-sample → multi-sample → merged → integrated). An optional deep-dive for pipeline authors.
